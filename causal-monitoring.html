Causal method deep dives — all 6 methods

1. Granger Causality — Temporal Precedence

Does knowing X’s past improve prediction of Y beyond Y’s own past? Use case: Do attendance dips Granger-cause grade declines? VAR model with F-test on lagged coefficients, AIC/BIC lag selection with max lag of 6 grading periods. Output: directed temporal influence graph with p-values. Limitation: establishes temporal precedence, not true causation — used as a screening tool.

2. Propensity Score Matching (PSM)

Match treated and untreated learners on observable characteristics. Use case: Causal effect of tutoring on math scores. Nearest-neighbor matching with caliper = 0.2σ of logit propensity. Balance check: standardized mean differences <0.1 on all covariates. Estimand: ATT. Sensitivity: Rosenbaum bounds for hidden bias. Covariates include 48+ features from the feature store.

3. Instrumental Variables (IV)

Use an instrument that affects treatment but not outcome directly. Use case: Causal effect of absences on achievement (weather as instrument). Two-stage least squares: regress attendance on instrument + controls, then outcome on predicted attendance. Validity: first-stage F > 10, exclusion restriction argument. Example instruments: weather severity, distance to school, lottery-assigned programs.

4. Difference-in-Differences (DiD)

Compare pre/post changes between treatment and control groups. Use case: Impact of new reading curriculum adopted by some schools. Assumption: parallel trends in pre-treatment period. Validation: placebo test with pseudo-treatment dates. Extensions: staggered DiD (Callaway-Sant’Anna) for rollout programs. System automatically tests parallel trends with event-study plots.

5. Bayesian Networks — Structural Causal Models

Learn directed acyclic graphs (DAGs) representing causal relationships. Use case: Map full causal structure among attendance, behavior, grades, and home factors. Structure learning via PC algorithm + domain expert priors. Parameter learning via Bayesian estimation with Dirichlet priors. Inference: do-calculus for interventional queries P(Y | do(X)). Unlike single-equation methods, captures the entire web of causal relationships simultaneously.

6. Emergent Pattern Discovery

Unsupervised detection of causal chains across the knowledge graph. The system traverses relationships between entities — bus routes, attendance, instruction time, assessment scores — and applies Granger + Bayesian methods at each edge to validate causality. Discovers multi-hop chains like the bus route example: no human configured these connections. Runs nightly as a batch job, surfacing new discoveries ranked by effect size and population affected.

Causal methods — mathematical formulations

Granger Causality

For every (X,Y) time-series pair, test with maxlag=4 weeks: H₀: Past values of X do NOT improve prediction of Y If p < 0.05 → directed temporal edge: (X) -[:GRANGER_CAUSES {lag, p}]→ (Y)

PSM: Average Treatment Effect on Treated

ATT = (1/N_T) Σ_{i∈T} (Y_i^T - Y_{j(i)}^C) j(i) = matched control for treated student i Caliper = 0.1 on propensity score Robustness: placebo treatment refutation (should yield ~0)

Difference-in-Differences

ATT = [E(Y|G=1,T=1) - E(Y|G=1,T=0)] - [E(Y|G=0,T=1) - E(Y|G=0,T=0)] G=1: treated group (affected by policy change) G=0: control group (not affected) T=0: pre-period, T=1: post-period

Playbook engine — automated workflows

A Playbook is a district-configured workflow template: “When [trigger], execute [sequence], with [escalation rules].”

Example: Chronic Absence Early Intervention (PB-ATTENDANCE-001)

Trigger: unexcused_absence_count_30d ≥ 3 (with 30-day cooldown) Step 1: Create case (type: attendance_concern, assign: attendance coordinator) Step 2: Generate family communication (template, preferred language) — REQUIRES APPROVAL Step 3: Create task “Follow-up phone call” (due: 5 days) Step 4: Wait 14 days → evaluate attendance_rate_14d → If improved (≥90%): resolve case → If not: escalate to SST referral Step 5: If SST needed: schedule meeting, gather data packet, invite team

Monitoring & Closed Loop

Detect → Act → Monitor → Adapt → Learn

ATLAS doesn’t just recommend interventions — it tracks whether they’re working and automatically triggers re-evaluation when they’re not. This is the fundamental differentiator from every dashboard product on the market.

The Sarah story: ATLAS detected Sarah’s reading progress stalled at Week 8 — 3 weeks before her teacher noticed. The system checked fidelity (yes, being delivered), checked dosage (yes, attending), then triggered a futility gate: “Adequate fidelity + adequate dosage + no response = change intervention.” A new evidence-matched program was recommended, approved by the case manager, and Sarah’s trajectory recovered.

1. Fidelity Gate

Is the intervention being delivered as designed? If not, fix delivery before changing the intervention itself.

2. Dosage Gate

Is the student receiving enough time? If attendance or scheduling issues reduce dosage below 80%, fix scheduling first.

3. Futility Gate

Past midpoint with adequate fidelity and dosage but no response? Change intervention. The current approach isn’t working.

4. Positive Divergence

Exceeding expectations? Consider step-down to lower tier. Free up intensive resources for students who need them more.

CUSUM divergence detection — mathematical formulation

Cumulative Sum Control Chart

Parameters: k = 0.5 (allowance), h = 4.5 (decision interval) Upper accumulator (exceeding): S⁺_t = max(0, S⁺_{t-1} + (x_t - μ₀ - k)) Lower accumulator (falling): S⁻_t = max(0, S⁻_{t-1} + (μ₀ - k - x_t)) Alert when: S⁺_t > h (positive divergence) OR S⁻_t > h (negative divergence) Properties: - ~1 false alarm per year with weekly monitoring - Detects 0.5σ shifts within 8-12 observations - Separate accumulators for exceeding and falling behind

Bayesian Growth Projection

Conjugate Normal-Normal model with population prior from intervention evidence base (tighter for Tier 1, wider for Tier 4). Individual posterior updates as each progress monitoring data point arrives. Confidence bands widen with time. The projection is a living entity updated with every observation.

Impact Ledger & Learning Loop

The Impact Ledger is an immutable record linking every resource allocation decision to its measured outcome. Each entry tracks: the action plan, the decision provenance chain, resources consumed (staff hours, direct cost, materials, facilities, training, opportunity cost — fully loaded via Ingredients Method), expected outcomes (metric, baseline, target, projected), actual outcomes (as monitoring data arrives), and computed eROI.

The Learning Loop

Every completed intervention cycle produces an OutcomeLearningRecord that:

1. Updates the intervention’s effectiveness estimates for this learner profile cluster. 2. Improves the risk prediction model’s feature importance weights. 3. Refines the matching algorithm’s Thompson Sampling distributions. 4. Adds to the evidence base for similar-case retrieval. 5. Feeds the federated learning pipeline (cross-district knowledge transfer).

Compounding intelligence: The system gets measurably smarter with every intervention cycle. This is the data moat — each cycle of recommend → deliver → measure → learn makes the next recommendation better.
