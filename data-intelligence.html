Data Integration

From messy spreadsheets to clean knowledge

Raw data arrives from district systems in wildly inconsistent formats — CSV exports from 1990s platforms, REST APIs from modern SIS, XML dumps from SPED tools, manually entered spreadsheets. ATLAS transforms it all into interconnected ontology objects.

Entity resolution — the Splink pipeline

This is the most critical and error-prone step. When NWEA MAP has “Johnathan Smith” and the SIS has “John Smith Jr.” and the SPED system has “Johnathan A. Smith” — are these the same student? Getting this wrong means a student’s assessment data disconnects from their IEP.

Blocking Strategy

Without blocking, 50K students × 50K = 2.5 billion comparisons. Blocking rules reduce to plausible candidate pairs:

# Rule 1: Exact DOB + first 3 chars of last name l.date_of_birth = r.date_of_birth AND substr(l.last_name,1,3) = substr(r.last_name,1,3) # Rule 2: Exact last name + first 3 chars first name + same school l.last_name = r.last_name AND substr(l.first_name,1,3) = substr(r.first_name,1,3) AND l.school_id = r.school_id # Rule 3: Exact district student ID (strongest when available) l.district_student_id = r.district_student_id AND l.district_student_id IS NOT NULL # Rule 4: Soundex of last name + DOB year + gender (phonetic variations) soundex(l.last_name) = soundex(r.last_name) AND substr(l.dob,1,4) = substr(r.dob,1,4)

Comparison Columns (Fellegi-Sunter)

Each field is compared using appropriate string similarity: Jaro-Winkler for names (handles “John” vs “Johnathan”), exact match for DOB (strongest signal), date-diff with tolerances for data entry errors, term frequency adjustments (rare names provide stronger match evidence).

Match Thresholds

ProbabilityAction
≥ 0.95Auto-link (no human review needed)
0.70 – 0.95Send to human review queue
< 0.70Auto-reject (create as new entity)

Survivorship Rules (Golden Record)

When records match, which source wins for each field? SIS is authoritative for identity/enrollment fields. SPED system is authoritative for support plan data. Assessment platforms are authoritative for assessment events. Program flags use union strategy (merge from all sources).

Signal detection — 5 methods, 48+ features

Once data flows into the graph, the Signal Detection Service continuously monitors every learner across 48+ computed features. Five complementary detection methods run in parallel:

1. Threshold-Based

Simple but effective: attendance below 90%, GPA below 2.0, more than 3 referrals in 30 days. Configurable per district via InstitutionalContext.

2. Trend Detection

Detects trajectory changes using regression analysis. A student dropping from B+ to C over 6 weeks triggers before they hit any absolute threshold.

3. Anomaly Detection

Statistical outlier detection relative to the student’s own baseline and peer cohort. Uses Isolation Forest and z-score methods.

4. Convergent Signal

The most powerful method: detects when multiple weak signals across different domains converge. Grade dip + attendance drop + behavior referral = high-confidence alert even when no single metric triggers.

5. GNN-Based (Graph Neural Network)

Leverages the graph structure itself. Uses GraphSAGE embeddings to detect when a student’s neighborhood pattern resembles students who previously needed intervention.
