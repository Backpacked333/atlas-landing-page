Education Operating System

From 47 disconnected systems
to one intelligence layer

ATLAS is a Palantir-grade semantic reasoning platform for K-12 education. It ingests messy data from every school system, reasons about what it finds, and recommends actions — with humans always in control.

5 Storage Engines 12 Microservices 9 Operational Domains 12 Optimization Models 48+ Learner Features 5 Causal Methods
Why does K-12 education need an operating system?

A typical school district operates 47 disconnected software systems — one for student information, another for assessments, another for special education, another for scheduling, another for behavior tracking, another for bus routing. Data lives in silos. A counselor wondering “Is this student at risk?” must manually check 6 different systems.

The consequences are real: IEP service minutes go undelivered because the scheduling system doesn’t know about the SPED system’s requirements. A student’s reading intervention isn’t working, but nobody notices for months because the progress monitoring data doesn’t connect to the intervention assignment system. A bus route change causes a student to arrive late to math — their fractions scores decline, but no system connects the cause to the effect.

ATLAS solves this by creating a unified knowledge graph that connects everything: every student, teacher, course, assessment, intervention, schedule, bus route, and IEP obligation. Then it reasons about those connections using the same algorithmic approaches that transformed logistics, finance, and healthcare.

Five architectural principles: (1) The ontology is the API — no direct database queries. (2) Events are the source of truth — current state is a materialized view. (3) Security is structural — access control on every graph traversal. (4) Intelligence is layered — deterministic algorithms first, ML/LLM augments. (5) Human authority is absolute — the system proposes, humans decide.

System Architecture

Five layers. One reasoning substrate.

Polyglot persistence with an event-driven microservices architecture. Five storage engines, each chosen for specific query patterns, unified by a semantic layer that presents a single model to application code.

Storage engines — why five databases?

No single database handles graph traversal, full-text search, bitemporal queries, OLTP transactions, and sub-millisecond caching equally well. ATLAS picks the best tool for each job:

EngineRoleWhy This One
Neo4j Enterprise 5.x Graph store — relationships, 6+ hop traversals, GDS algorithms Index-free adjacency gives 10-50ms for multi-hop queries. Composite Databases for per-district sharding. 65+ GDS algorithms (Leiden, PageRank, FastRP embeddings).
PostgreSQL 16.x Transactional OLTP — accounts, config, constraint registry ACID guarantees. Row-Level Security for tenant isolation. JSONB for flexible schema. SQL:2011 temporal tables.
XTDB v2 Bitemporal compliance — point-in-time reconstruction, audit Native bitemporality (valid-time + system-time) on ALL tables. Postgres wire protocol. Hot/cold storage via S3.
Elasticsearch 8.x Search — full-text, faceted filtering, autocomplete Sub-50ms full-text search. Nested object queries. Index-per-tenant isolation.
Redis 7.x Cache — results, sessions, rate limiting, pub/sub Sub-1ms reads. Pub/sub for event-driven cache invalidation.

Event backbone — Kafka architecture

Apache Kafka (KRaft mode, 3.7+) serves as the central nervous system. Every state change produces events — CDC streams from source systems, domain events from mutations, audit events from access, provenance events from reasoning.

Communication Patterns

Synchronous (GraphQL/gRPC): Client → API Gateway → Query/Action Service for user-facing operations. Service-to-service for constraint validation.

Asynchronous (Kafka): CDC from source systems → Ingestion Service. Domain events → all consumers. Signal detected → Case & Workflow. Provenance → Audit + XTDB.

Multi-tenancy — per-district isolation

Every piece of data carries a tenantId (format: org_{uuid}). The Query Service injects tenant filters at the query planning layer — application code never constructs unfiltered queries.

EngineIsolation Method
Neo4jComposite Database with per-tenant constituent databases
PostgreSQLRow-Level Security with tenant_id policies
KafkaTenant-prefixed topics
ElasticsearchIndex-per-tenant with filtered aliases
RedisKey prefixing ( org_abc123:cache:learner:456 )
XTDBTenant column with query-time filtering

Institutional Context: Each tenant has an InstitutionalContext configuration driving which constraints, entity types, assessments, and reasoning behaviors are active — serving a 200-student Montessori school and an 80,000-student urban district with the same architecture.

SEMANTIC ONTOLOGY

Everything connects to the student

Nine operational domains, hundreds of entities, thousands of relationships — all centered on the learner. The ontology isn’t a database schema. It’s a reasoning substrate where every traversal reveals insights no single system could produce alone.

📚

Instructional Program

Courses, sections, curricular frameworks (Common Core to Montessori), assessment instruments, materials, pacing guides. Framework-agnostic — models the concept of structured learning.

👥

People

Staff (teachers, counselors, therapists), guardians, households. Certifications, caseloads, contractual obligations, expertise areas. Every role relationship is a graph edge.

📊

Assessment

Every formal and informal assessment event — MAP, DIBELS, state tests, teacher-created, observational. Subdomain scores, accommodations, psychometric properties, validity flags.

📄

Support Plans

IEPs, 504 plans, intervention plans, gifted service plans, EL plans, BIPs. Goals, service obligations, accommodations, progress monitoring, compliance deadlines.

🕐

Time & Space

Bell schedules, calendar, facilities, rooms, capacities. Schedule blocks that connect courses to teachers to rooms to time slots.

💰

Finance

Budgets, funding sources (Title I, IDEA, grants), expenditure tracking, cost per student, program cost allocation via Ingredients Method.

💪

Intervention

Programs, evidence base ratings, assignment matching, dosage tracking, fidelity monitoring, outcome measurement.

💡

Behavioral & SEL

Behavior events (positive and negative), PBIS points, SEL assessments (DESSA, SAEBRS), engagement patterns, incident escalation tracking.

🧳

Governance

Decision provenance (reified reasoning subgraphs), audit trails, compliance status, regulatory requirements, constraint catalog.
