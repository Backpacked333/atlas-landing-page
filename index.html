Education Operating System

From 47 disconnected systems
to one intelligence layer

ATLAS is a Palantir-grade semantic reasoning platform for K-12 education. It ingests messy data from every school system, reasons about what it finds, and recommends actions — with humans always in control.

5 Storage Engines 12 Microservices 9 Operational Domains 12 Optimization Models 48+ Learner Features 5 Causal Methods
Why does K-12 education need an operating system?

A typical school district operates 47 disconnected software systems — one for student information, another for assessments, another for special education, another for scheduling, another for behavior tracking, another for bus routing. Data lives in silos. A counselor wondering “Is this student at risk?” must manually check 6 different systems.

The consequences are real: IEP service minutes go undelivered because the scheduling system doesn’t know about the SPED system’s requirements. A student’s reading intervention isn’t working, but nobody notices for months because the progress monitoring data doesn’t connect to the intervention assignment system. A bus route change causes a student to arrive late to math — their fractions scores decline, but no system connects the cause to the effect.

ATLAS solves this by creating a unified knowledge graph that connects everything: every student, teacher, course, assessment, intervention, schedule, bus route, and IEP obligation. Then it reasons about those connections using the same algorithmic approaches that transformed logistics, finance, and healthcare.

Five architectural principles: (1) The ontology is the API — no direct database queries. (2) Events are the source of truth — current state is a materialized view. (3) Security is structural — access control on every graph traversal. (4) Intelligence is layered — deterministic algorithms first, ML/LLM augments. (5) Human authority is absolute — the system proposes, humans decide.

System Architecture

Five layers. One reasoning substrate.

Polyglot persistence with an event-driven microservices architecture. Five storage engines, each chosen for specific query patterns, unified by a semantic layer that presents a single model to application code.

Event backbone — Kafka architecture

Apache Kafka (KRaft mode, 3.7+) serves as the central nervous system. Every state change produces events — CDC streams from source systems, domain events from mutations, audit events from access, provenance events from reasoning.

Communication Patterns

Synchronous (GraphQL/gRPC): Client → API Gateway → Query/Action Service for user-facing operations. Service-to-service for constraint validation.

Asynchronous (Kafka): CDC from source systems → Ingestion Service. Domain events → all consumers. Signal detected → Case & Workflow. Provenance → Audit + XTDB.

Temporal modeling — bitemporal everything

ATLAS maintains two time dimensions for every fact using XTDB v2:

Valid-time: When a fact was true in the real world (e.g., student enrolled on Sept 1).

System-time: When the system learned about it (e.g., data ingested on Sept 3).

This enables "as-of" queries: "What did the system know about this student on October 15?" — critical for FERPA compliance, audit trails, and understanding how decisions were made with the information available at the time.

-- XTDB temporal query: what was true at a specific moment
SELECT * FROM learner_enrollment
FOR VALID_TIME AS OF DATE '2025-10-15'
FOR SYSTEM_TIME AS OF TIMESTAMP '2025-10-16T09:00:00Z'
WHERE learner_id = 'abc123'

Entity resolution — the Splink pipeline

This is the most critical and error-prone step. When NWEA MAP has "Johnathan Smith" and the SIS has "John Smith Jr." and the SPED system has "Johnathan A. Smith" — are these the same student? Getting this wrong means a student’s assessment data disconnects from their IEP.

Blocking Strategy

Without blocking, 50K students × 50K = 2.5 billion comparisons. Blocking rules reduce to plausible candidate pairs:

# Rule 1: Exact DOB + first 3 chars of last name
l.date_of_birth = r.date_of_birth AND substr(l.last_name,1,3) = substr(r.last_name,1,3)

# Rule 2: Exact last name + first 3 chars first name + same school
l.last_name = r.last_name AND substr(l.first_name,1,3) = substr(r.first_name,1,3) AND l.school_id = r.school_id

# Rule 3: Exact district student ID (strongest when available)
l.district_student_id = r.district_student_id AND l.district_student_id IS NOT NULL

# Rule 4: Soundex of last name + DOB year + gender (phonetic variations)
soundex(l.last_name) = soundex(r.last_name) AND substr(l.dob,1,4) = substr(r.dob,1,4)

Comparison Columns (Fellegi-Sunter)

Each field is compared using appropriate string similarity: Jaro-Winkler for names (handles "John" vs "Johnathan"), exact match for DOB (strongest signal), date-diff with tolerances for data entry errors, term frequency adjustments (rare names provide stronger match evidence).

Match Thresholds

ProbabilityAction
≥ 0.95Auto-link (no human review needed)
0.70 – 0.95Send to human review queue
< 0.70Auto-reject (create as new entity)

Survivorship Rules (Golden Record)

When records match, which source wins for each field? SIS is authoritative for identity/enrollment fields. SPED system is authoritative for support plan data. Assessment platforms are authoritative for assessment events. Program flags use union strategy (merge from all sources).

When a signal creates a case, the LangGraph agent orchestrates a multi-step reasoning process:

Graph-Constrained Reasoning (ICML 2025): The LLM’s reasoning is constrained by the knowledge graph structure — it can only reference entities and relationships that actually exist. This prevents hallucination and ensures every claim is grounded in data.

Think-on-Graph & Graph-RAG

Think-on-Graph (ICLR 2024): Instead of treating the KG as a static context dump, the LLM iteratively explores the graph through beam search. At each step, it selects the most promising edges to traverse based on its reasoning needs.

Graph-RAG: Extract a relevant subgraph around the case entities (typically 2-3 hops), serialize it, and provide it as grounded context to the LLM. The LLM can then reference specific nodes and edges in its reasoning.

Tool Registry (14 tools): The agent has governed tools for ontology queries, constraint checking, optimization invocation, evidence base search, explanation generation, and more. Each tool call is logged with full provenance.

Master scheduling — mathematical formulation

Decision Variables

x[c,t,r] ∈ {0,1} : course section c scheduled at time slot t in room r

Objective

min Σ_h w_h · violation_h + Σ_s w_s · violation_s

Hard Constraints

Teacher conflicts (one location per slot), room conflicts (one section per room per slot), room capacity (enrollment ≤ capacity), student conflicts (no simultaneous sections), IEP service minutes (pull-out blocks must exist), teacher certification (valid cert for subject).

Soft Constraints (weighted)

Student preference satisfaction, teacher schedule compactness, equitable access to advanced courses, pedagogical sequencing (lab follows lecture), planning period distribution per contract.

Solver: Google OR-Tools CP-SAT with hot-start from prior year schedule. Decomposed into teacher assignment + room assignment + student scheduling sub-problems. Large Neighborhood Search for refinement.
