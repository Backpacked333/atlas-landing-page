Intelligence & Optimization

LangGraph-orchestrated LLM agents reason over the knowledge graph while 12 mathematical optimization models solve scheduling, routing, and resource-matching problems at district scale.

LLM Agent Layer

LangGraph agent orchestration

An 11-node state graph coordinates LLM reasoning with tool execution. The architecture enforces governance at every step — no unchecked model output reaches downstream systems.

Governed Tool Registry

8 tools, zero free-text SQL

Every tool the agent can invoke is whitelisted, schema-validated, and rate-limited. The LLM never writes raw queries — it selects parameterized operations from a governed registry.

Reasoning Patterns

Two core system prompts

Specialized system prompts guide the LLM through structured reasoning chains, ensuring consistent and auditable analysis.

Root-Cause Analysis Prompt

The root-cause analysis prompt guides the agent through a 5-step reasoning chain:

Step 1: Signal Decomposition

Break down the composite signal into contributing features with attribution weights.

Step 2: Temporal Analysis

Examine feature trajectories over time. Identify inflection points and rate-of-change anomalies.

Step 3: Graph Context

Traverse the knowledge graph to find related entities, peer comparisons, and environmental factors.

Step 4: Hypothesis Formation

Generate ranked hypotheses with confidence scores, citing specific data points as evidence.

Step 5: Intervention Matching

Map each hypothesis to evidence-based interventions from the catalog, computing eROI for each.

Intervention Matching Prompt

The intervention matching prompt ensures recommendations are feasible, evidence-based, and equitable:

Constraint Awareness

Check staff availability, budget remaining, caseload limits, and scheduling conflicts before recommending.

Evidence Hierarchy

Prefer interventions with Tier 1 (strong) or Tier 2 (moderate) evidence from What Works Clearinghouse.

Equity Check

Verify proportional representation across demographic groups. Flag disparate impact exceeding threshold.

eROI Ranking

Rank all viable interventions by expected ROI, presenting top 3 with confidence intervals and trade-offs.

Solver Engines

Four solvers, one dispatch layer

Each solver is containerized with standardized input/output schemas. The dispatch layer handles serialization, timeout management, and solution caching.

OR-Tools CP-SAT

Google’s constraint programming solver — the workhorse for ATLAS scheduling problems.

  • Handles 6 of the 12 models (master schedule, student matching, section balancing, assessment calendar, space utilization)
  • Supports Boolean, integer, and interval variables with global constraints
  • Parallel search with portfolio of strategies (default, fixed, random)
  • Typical solve time: < 30s for a 2,000-student school’s master schedule
  • Warm-start from previous solutions for incremental re-optimization
Gurobi Optimizer

Commercial-grade MILP solver for large-scale linear and integer programming.

  • Budget allocation with 100+ constraint rows per district
  • Multi-year capital planning with stochastic demand scenarios
  • Bus routing sub-problems when OR-Tools hits scalability limits
  • Lazy constraints for iterative cut generation
  • Cloud-hosted with per-district license isolation
Timefold (OptaPlanner)

Java-based constraint solver optimized for dynamic planning with frequent re-scheduling.

  • Substitute teacher assignment with real-time availability updates
  • Professional development scheduling with rolling enrollment
  • Late-acceptance and tabu search metaheuristics
  • Continuous planning mode — re-optimizes as constraints change
  • Score director provides incremental constraint evaluation
Z3 SMT Solver

Microsoft’s Satisfiability Modulo Theories solver for logical constraint verification.

  • Enrollment projection feasibility under policy change scenarios
  • Validates that proposed schedules satisfy all hard constraints before deployment
  • Unsat core extraction pinpoints the minimal set of conflicting constraints
  • Theory combination: linear arithmetic + bit vectors + arrays
  • Used as a verification layer after heuristic solvers produce candidate solutions

Expected Return on Investment

The eROI engine

Every recommendation carries an expected ROI score computed from historical outcomes, implementation costs, and opportunity costs. This moves education decisions from intuition to evidence.

P(s)

Success Probability

LightGBM-predicted likelihood of positive outcome given learner features and intervention type. Calibrated via Platt scaling.

ΔO

Expected Outcome Gain

Projected improvement in target metric (attendance points, grade-level proficiency, behavior incidents avoided).

Ct

Total Cost

Staff time + materials + opportunity cost of displaced alternatives. Amortized over the intervention duration window.

eROI Formula

eROI = P(s) × ΔO ⁄ Ct

Confidence interval derived from bootstrap resampling of historical intervention outcomes (n ≥ 30 required).

Portfolio Optimization

Given a fixed budget, the system selects the portfolio of interventions that maximizes total eROI across all flagged learners, subject to staff and scheduling constraints.

Feedback Loop

Actual outcomes feed back into the eROI model quarterly. Interventions that consistently underperform their predicted eROI are flagged for review and potential deprecation.

Constraint Engine

Guardrails that never bend

Hard constraints encoded in Z3, Drools, and OR-Tools ensure that no optimization output violates legal, policy, or safety requirements.

Legal Constraints

  • FERPA data access boundaries
  • IDEA service minute minimums
  • Title I funding allocation rules
  • State certification requirements

Policy Constraints

  • Maximum class size by grade band
  • Minimum planning periods for teachers
  • Consecutive period teaching limits
  • Equity distribution thresholds

Operational Constraints

  • Bus route time windows
  • Room feature requirements (ADA, lab)
  • Staff travel time between sites
  • Budget ceiling enforcement
Unsat Core Reporting: When constraints conflict and no feasible solution exists, the Z3 solver extracts the minimal unsatisfiable core — the smallest set of constraints causing the conflict. This is presented to administrators in natural language so they can decide which soft constraint to relax.
