Access Control

ABAC + ReBAC: dual-layer authorization

Attribute-Based Access Control defines what you can see based on your role and attributes. Relationship-Based Access Control refines it based on your actual relationships to students and schools.

ABAC Layer

OPA Rego policies evaluate requests against user attributes, resource attributes, and environmental context.

allow { input.user.role == "teacher" input.resource.type == "learner_record" input.user.district == input.resource.district is_active_school_year(input.context.date) }
  • Role hierarchy: district admin > principal > counselor > teacher > para
  • Temporal scoping: access expires with employment and school year
  • Data classification levels: public, internal, confidential, restricted

ReBAC Layer

Graph-based relationship checks ensure users can only access data for students they have a legitimate educational relationship with.

has_relationship { # Teacher → teaches → Section → enrolls → Student path_exists(input.user, "teaches", "section", "enrolls", input.resource.student) }
  • Relationship types: teaches, counsels, administers, parents
  • Transitive closure for principal → school → all students
  • Emergency access with mandatory justification logging
Defense in Depth: Both ABAC and ReBAC must pass for any data access. A teacher with the right role attributes but no relationship to a student is denied. A related user without proper attributes is also denied. This dual-gate eliminates entire classes of data leakage.

Data Protection

Encryption & crypto-shredding

Per-district encryption keys with hardware-backed key management. When a district leaves, crypto-shredding ensures complete, verifiable data destruction.

Encryption Architecture
  • At rest: AES-256-GCM with per-district DEKs (Data Encryption Keys)
  • In transit: TLS 1.3 with certificate pinning for service-to-service
  • Key hierarchy: AWS KMS master key → district KEK → table-level DEK
  • Rotation: Automatic annual key rotation with re-encryption pipeline
  • HSM backing: FIPS 140-2 Level 3 validated hardware security modules
  • Field-level: SSN, medical records encrypted at field level with separate keys
Crypto-Shredding Process
  • Trigger: District contract termination or right-to-delete request
  • Step 1: Revoke district KEK from KMS (immediate access cut-off)
  • Step 2: Queue async deletion of all encrypted data blocks
  • Step 3: Purge derived data (features, embeddings, model weights)
  • Step 4: Generate cryptographic proof of deletion (Merkle tree audit)
  • Step 5: Third-party attestation report delivered to district

Audit & Logging

Immutable audit trail

Every data access, model prediction, and recommendation is logged with full provenance. Tamper-evident logs support compliance audits and incident investigation.

Log TypeContentsRetentionStorage
Access LogWho accessed what record, when, from where, why7 yearsImmutable append-only (S3 Object Lock)
Decision LogModel inputs, outputs, confidence scores, explanation chain7 yearsXTDB bitemporality (queryable)
Change LogAll data mutations with before/after snapshots7 yearsKafka + XTDB event store
Auth LogLogin attempts, token grants, permission evaluations3 yearsElasticsearch (SIEM integration)
System LogService health, errors, performance metrics1 yearOpenTelemetry → Grafana

Tamper Evidence

Log entries are hash-chained (SHA-256) with periodic Merkle root publication. Any modification to historical entries breaks the chain, making tampering immediately detectable.

Real-Time Alerting

Anomalous access patterns (bulk exports, off-hours access, unusual query scope) trigger real-time alerts to security team via PagerDuty integration.
