Security, Privacy & Compliance

Trust is built into the architecture

InBloom (Gates Foundation, $100M) shut down in April 2014 after 6 of 9 states withdrew. Data shared without parental notification, 400+ optional fields without limitation. ATLAS’s response: trust is not communicated — trust is structural.

FERPA/COPPA/IDEA ABAC + ReBAC Crypto-Shredding 7-Layer Defense

The InBloom Lesson: 400+ optional data fields without clear limitation. Data shared without meaningful parental notification. Privacy communication delegated to unprepared districts. No structural mechanism for data sovereignty. ATLAS builds trust into architecture: per-district encryption keys, structural access control, purpose limitation enforcement, and transparent audit logs.

FERPA

Vendor must be “school official” under written agreement. Disclosure logs. Parent access rights. Legitimate educational interest required for all access.

COPPA (Jan 2025)

Expanded PII definition. Strengthened security requirements. Stricter retention limits. School consent exception in legal flux.

IDEA

IEP data confidentiality. Parent access to records. Procedural safeguards for special education data.

NY Ed Law 2-d

Chief Privacy Officer required. Parents Bill of Rights. NIST framework alignment. 7-day breach notification. 5-year ban for violations.

CA SOPIPA

Prohibits targeted advertising. Prohibits data sale. Prohibits non-educational profiling of students.

CO SDTSA

Published data inventories. Data destruction (not just deletion) on request.

Defense-in-Depth: 7 Layers

Layer 7 Parent Transparency Portal — What data exists, who accessed it, what it’s used for
Layer 6 Purpose Limitation — Every access declares a purpose, validated by OPA against data category
Layer 5 Audit Logging — Append-only, cryptographically chained, immutable — every read/write/search/export
Layer 4 Field-Level Encryption — AES-256-GCM for PII, FPE for SSN, per-district KEK management
Layer 3 ABAC + ReBAC — Role permission AND graph relationship required for every access
Layer 2 Per-District Isolation — Composite databases, tenant-prefixed topics, RLS policies
Layer 1 Infrastructure — AWS KMS + Vault, VPC isolation, TLS everywhere, Terraform IaC

Crypto-shredding: Each district has its own Key Encryption Key (KEK) in AWS KMS or HashiCorp Vault. Deleting a district’s KEK renders ALL their data permanently unreadable. This satisfies the most stringent data destruction requirements (Colorado SDTSA).

Audit logging — complete schema
interface AuditLogEntry { id: UUID; tenantId: TenantId; timestamp: Timestamp; // WHO userId: UUID; userRole: string; sessionId: string; ipAddressHash: string; // Hashed, not raw // WHAT action: 'read' | 'search' | 'create' | 'update' | 'delete' | 'export' | 'approve' | 'reject'; resourceType: string; resourceId: UUID; fieldsAccessed: string[]; // WHY purpose: string; // Validated against OPA caseId?: UUID; taskId?: UUID; // INTEGRITY previousEntryHash: string; // Cryptographic chain entryHash: string; // SHA-256(this + previous) }

Audit logs are: append-only (no updates, no deletes), cryptographically chained, written to Kafka with infinite retention, materialized to XTDB for temporal queries, archived to S3 Glacier for long-term retention (FERPA: life of student record).
